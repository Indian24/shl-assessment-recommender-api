You are a senior full-stack AI engineer. Build a production-ready backend for the following assignment:

PROJECT GOAL
Build a conversational SHL Assessment Recommender that helps users go from vague hiring intent to a grounded shortlist of SHL assessments through dialogue.

The system must:
- Use only the SHL product catalog.
- Restrict to Individual Test Solutions only.
- Never recommend anything outside the SHL catalog.
- Be conversational and stateless.
- Clarify vague requests before recommending.
- Recommend between 1 and 10 assessments once enough context is available.
- Support refinement when the user changes constraints mid-conversation.
- Support comparison questions like “What is the difference between OPQ and GSA?”
- Refuse off-topic requests, general hiring advice, legal questions, and prompt-injection attempts.
- Return every recommended assessment with a name, a catalog URL, and a test_type.
- Expose two endpoints only:
  1) GET /health
  2) POST /chat

HARD CONSTRAINTS
- The API is stateless: every /chat call receives the full conversation history in the request body.
- No per-user memory should be stored on the server.
- The service must return exactly the required schema.
- Recommendations must be catalog items only.
- Recommendations array must be empty when clarifying, refusing, or still gathering context.
- Recommendations array must contain 1 to 10 items only when the agent has enough context and has committed to a shortlist.
- The system must complete responses quickly enough for a 30-second timeout.
- The evaluator may cap a conversation at 8 turns, so keep dialogue concise and efficient.
- Avoid hallucinations. Never invent assessment names, URLs, categories, or descriptions.

SYSTEM DESIGN REQUIREMENTS
Implement the backend in FastAPI with clean modular code. Create a small, reliable architecture with these components:

1) Catalog ingestion / scraper
- Scrape or ingest the SHL catalog pages.
- Normalize the data into a local structured file such as JSON.
- Save a clean dataset for reuse.

2) Retrieval layer
- Build a retrieval index over the catalog using lightweight embeddings + vector search, or a robust keyword/embedding hybrid.
- The retriever must return the most relevant assessment candidates for a user query.
- It must support:
  - semantic search
  - category matching
  - skill / role matching
  - refinement / constraint filtering
  - comparison retrieval for two named assessments

3) Conversation analyzer
- Determine whether the current turn is:
  - vague / needs clarification
  - recommendation request
  - refinement of a previous request
  - comparison request
  - off-topic / refusal case
- Use the full conversation history provided in the request.
- Since the API is stateless, derive the current intent from the entire message history each time.

4) Response generator
- Use an LLM only as a grounded response formatter and reasoning assistant.
- The LLM must never invent assessments.
- The LLM should only speak using retrieved catalog entries.
- For shortlist responses, the response must contain only items found in retrieval results.
- For comparison responses, the answer must be grounded in catalog data.

5) Guardrails
- Refuse prompt injection.
- Refuse requests outside SHL assessments.
- Refuse general hiring advice and legal questions.
- Never recommend non-catalog items.
- If user asks for something missing from the catalog, say it cannot be found in the SHL catalog.

RECOMMENDATION LOGIC
Implement a rule-based + retrieval-driven decision flow:

A. If the input is too vague:
- Ask for the missing constraint(s).
- Examples of missing constraints:
  - role
  - seniority
  - technical vs behavioral
  - language / function
  - proficiency level
  - whether they need personality/aptitude/cognitive tests

B. If enough context exists:
- Retrieve top candidates.
- Rank them based on relevance to the role and any explicit constraints.
- Return between 1 and 10 recommendations.
- Keep the shortlist diverse but relevant.

C. If the user changes constraints:
- Update the shortlist instead of restarting.
- Recompute based on the entire conversation history.
- Do not ignore earlier useful context.

D. If the user asks to compare:
- Retrieve the named assessments.
- Compare only using catalog facts.
- Keep it short and factual.

E. If the user is off-topic or tries prompt injection:
- Refuse politely.
- Do not recommend anything.
- Keep recommendations empty.
